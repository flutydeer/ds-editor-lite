# 工程资源跨设备可移植性方案

> 状态：讨论后修订 v2（2026-07-17）
> 已定：存储机制对齐 DiffScope，直接读写 `workspace["diffscope"]["audio"]`（§2）；FileContentChanged 静默处理（§3.1）；缺失处理扩展为统一资源检查，覆盖音频与歌手包（§3.3）。

## TL;DR

**问题**：音频 clip 在工程里存的是导入时的**绝对路径**，创作者把工程连同音频分享给别人后，对方机器上路径必然失效，音频加载不出来，而且没有任何补救入口。歌手/声线包同理——对方设备可能没装对应的包。

**方案**：

1. 工程文件存储对齐 DiffScope：`path` 字段**保持绝对路径**（生态兼容），定位信息（absoluteDir / relativeDir / fileName / sha512）存入 **`workspace["diffscope"]["audio"]`**，与 DiffScope 直接互认；加载时多级解析（绝对路径 → 工程目录 + 相对路径 → 工程同级同名，后两者需 hash 校验）；
2. 导入音频时**复制到工程资产目录**（如 `assets/audio/`），分享时拷贝「工程 + 资产目录」即可跨设备打开；
3. 自动解析成功即**静默**；剩余需要用户决策的项（音频找不到、歌手包未安装）进**统一的资源检查对话框**，支持音频重新定位与歌手重映射。

## 1. 问题根源

当前实现中，音频路径的处理链路如下：

- `AudioClip` 只保存一个路径字符串 `m_path`（`src/app/Model/AppModel/AudioClip.h`），导入时直接写入用户选择的**绝对路径**；
- 保存工程时原样序列化进 .dspx 的 `path` 字段（`DspxProjectConverter.cpp` 序列化处），加载时原样读回；
- 解码失败时没有"缺失媒体"的概念：clip 不会进入可恢复的离线状态，也没有重新定位（relink）的入口。

因此只要换一台机器（或移动音频文件），绝对路径必然失效，且用户没有任何补救手段。

资源缺失实际上不止音频一类：

| 资源 | 引用方式 | 缺失原因 | 修复手段 |
| --- | --- | --- | --- |
| 音频文件 | 文件路径 | 路径失效 / 文件被移动 | 自动解析回退、重新定位 |
| 歌手/声线包 | `SingerIdentifier { singerId, packageId, packageVersion }`（存于 workspace，见 `DspxProjectConverter.cpp` Identifier helpers） | 目标设备未安装对应包 | 安装包、重映射到已安装歌手 |

DAW 行业对此有成熟套路（Cubase「Back up Project」、Reaper「Save project + move media」、Premiere/Ableton 的缺失媒体重定位对话框）。同属 OpenVPI 生态的 **DiffScope** 已实现完整的音频路径定位机制，本方案的存储层直接对齐它。

## 2. 存储机制：对齐 DiffScope（已定)

参考实现：`diffscope-project` 的 `dspx::AudioPathInfo`（dspxmodel）、`AudioClipAddOn` 与 `ProjectAudioAddOn`（audio 插件）。

### 2.1 数据模型

音频 clip 的路径从一个字符串扩展为一组定位信息：

| 字段 | 说明 |
| --- | --- |
| `absoluteDir` | 音频文件所在目录的绝对路径 |
| `relativeDir` | 相对于**工程文件所在目录**的目录路径；仅当音频位于工程目录或其子目录内时填写，否则为空 |
| `fileName` | 文件名（目录与文件名分开存，重定位时只需替换目录部分） |
| `sha512` | 整个文件内容的 SHA-512 摘要，用于校验"找到的文件是不是原来那个" |

DiffScope 的 `AudioPathInfo` 另有 `formatEntryClassName` / `userData`（talcs 解码器恢复信息），ds-editor-lite 不引入这两个字段，但**读写时必须原样保留**（见 §2.2 要点）。

### 2.2 dspx 序列化布局（已定：直接使用 diffscope 命名空间）

**`path` 字段保持绝对路径**（`absoluteDir` + `fileName` 拼接），其他编辑器按规范读 `path` 不受任何影响；定位信息直接读写 **`workspace["diffscope"]["audio"]`**，与 DiffScope 保存的工程双向互认：

```jsonc
{
  "path": "C:/Users/xxx/Music/vocal.wav",
  "workspace": {
    "diffscope": {
      "audio": {
        "absoluteDir": "C:/Users/xxx/Music",
        "relativeDir": "assets/audio",    // 为空表示不在工程目录内
        "fileName": "vocal.wav",
        "sha512": "…"
        // DiffScope 可能还写有 formatEntryClassName / userData
      }
    }
  }
}
```

要点：

- **未知字段保留**：DiffScope 写入的 `formatEntryClassName` / `userData`（以及未来新增字段）在 ds-editor-lite 打开再保存后必须原样带回，不能丢失——反序列化时保留整个 `workspace["diffscope"]` 原始对象，仅更新我们认识的字段；
- **冗余一致性**：`absoluteDir` + `fileName` 与 `path` 字段冗余，保存时必须由同一来源生成，以 `path` 为规范值；
- DiffScope 当前的 reader 只读取 `relativeDir` / `formatEntryClassName` / `userData` / `sha512`，多出的 `absoluteDir` / `fileName` 字段会被其忽略（它从 `path` 解析这两项），互认无冲突；后续 schema 演进需与 DiffScope 侧保持沟通。

### 2.3 加载时的解析顺序（参考 DiffScope `ProjectAudioAddOn::loadAudioClip`）

1. **绝对路径存在** → 直接使用；若 sha512 不匹配 → **静默接受**（见 §3.1，此处与 DiffScope 有意不同）；
2. 绝对路径失效 → 尝试 **工程目录 + `relativeDir` + `fileName`**，文件存在**且 sha512 匹配** → 使用；
3. 再失效 → 尝试 **工程目录 + `fileName`**（工程同级兜底），同样要求 sha512 匹配；
4. 全部失败 → clip 进入缺失状态（§3.2）。

要点：

- 兜底路径（2、3）**必须 hash 校验**，避免同名不同文件误匹配；
- sha512 为空（工程来自不写该字段的编辑器）时，兜底命中只能按文件名匹配，标记为"待确认"进资源检查对话框（§3.3）；
- 回退命中后在内存中更新路径信息，**随下次保存自然落盘**（不主动写脏工程文件）——第二次打开即可绝对路径直接命中，系统自愈。

### 2.4 ds-editor-lite 的落地改动

- `AudioClip` 的 `m_path`（QString）扩展为 §2.1 的结构；
- `DspxProjectConverter` 按 §2.2 布局读写 `workspace["diffscope"]["audio"]`，并保证未知字段 round-trip；
- 导入音频时：若文件位于工程目录内则记录 `relativeDir`，并计算 sha512（后台任务，避免大文件阻塞 UI）；
- 加载音频时：按 §2.3 顺序解析，解析结果驱动 §3 的状态与交互。

## 3. 加载结果的分级交互

**总原则：只有需要用户决策的信息才打断用户。** hash 校验通过的自动定位在数学上就是原文件，"确认"不提供信息量；分享工程走 relativeDir 命中是设计内路径，不是异常。

### 3.1 分级表（音频）

| 解析结果 | 处理 |
| --- | --- |
| 绝对路径命中，hash 一致 | 正常加载 |
| 绝对路径命中，hash 不一致（内容变化） | **静默接受**：正常加载、重新解码峰值缓存、重算 sha512 随下次保存落盘。理由：路径没变而内容变了，几乎总是用户在本机主动替换（如换高音质版本），对齐主流 DAW 行为。*此处与 DiffScope 不同（它发警告通知）* |
| relativeDir 命中 + hash 匹配 | **完全静默**（分享工程的设计内路径）；内存更新路径，随保存回写 |
| 同级兜底命中 + hash 匹配 | 非阻断通知，全工程汇总一条："N 个音频已自动重新定位" |
| 兜底命中但 sha512 为空（无法校验） | 进资源检查对话框，标记"按文件名匹配，请确认" |
| 未找到 | 离线状态（§3.2）+ 进资源检查对话框 |

### 3.2 离线状态

- 音频 clip：轨道上显示占位（灰色 + 「文件缺失」标记），工程照常打开，clip 的时值/增益等信息完整保留；
- 歌手缺失的轨道：进入「歌手缺失」状态，禁用推理，**保留原 `SingerIdentifier` 不销毁**——用户之后安装了对应包，重新打开工程即自动恢复。

### 3.3 统一资源检查对话框（多页式）

音频和歌手包的缺失原因不同（路径失效 vs 包未安装），但对用户是同一个问题："这个工程在我的设备上缺东西"。因此用**一个对话框**统一处理，采用**安装向导式的多页交互**：每类资源一页，用户每次专注解决一类问题。

**范围**：v1 只实现**音频页**；歌手页排在以后，届时在既有骨架上加一页即可，对话框的页导航、汇总逻辑一次搭好。

**触发时机**：工程打开、所有资源解析完成后，**只在存在需要用户决策的项时弹出，且全工程只弹一次**；全部自动解决则不弹。菜单提供主动入口（如「工程 → 检查工程资源…」），点击离线 clip / 缺失歌手标记也可打开（直接定位到对应页）——静默不剥夺用户的控制权。

**多页交互要点**：

- **页面按需生成**：哪类资源没问题就不出现哪页；只有一类有问题时直接显示单页，不显示步骤导航；
- **非强制线性**：允许跳过未解决项进入下一页，完成时汇总"仍有 N 项未解决"；用户可随时关闭对话框，不阻塞工程打开，未解决项维持离线/缺失状态（§3.2）；
- 页顺序（歌手页加入后）：**歌手页在前**——歌手缺失影响推理能否运行，严重度高于只影响伴奏的音频缺失；
- 每页布局：该类资源的缺失项列表，每行：资源名、引用位置（哪些轨道/多少个 clip）、状态、操作。

**音频页操作**：

- **重新定位**：手动选择文件；定位成功一个后，自动以该文件所在目录尝试解析其余缺失项（hash 校验通过才采纳）——Premiere/Ableton 的经典做法；
- **保持缺失**：clip 维持离线状态。

**歌手/声线页操作**（以后实现；按歌手粒度，不逐轨道操作）：

- **安装对应包**：若包管理器能按 `packageId` 定位到包，引导下载安装——最优出路，工程不需要任何改动；
- **重映射**：下拉选择本机已安装的歌手，一次映射应用到所有引用该歌手的轨道；
- **保持缺失**：轨道维持「歌手缺失」状态（§3.2）。

**编辑语义与撤销边界（已定，v1 按此实现）**：undo 边界按"**内容是否可能变化**"划分——

- **手动重新定位（对话框内或右键菜单）是可撤销的工程编辑**：用户选文件时没有校验，完全可以选内容不同的文件（这正是换源功能），必须能反悔。进 undo 栈、置 dirty；
- **hash 校验通过的自动找回不进 undo 栈、不置 dirty**：包括工程打开时的多级解析命中（§2.3）与手动定位成功后的级联解析。sha512 匹配意味着找回的文件在数学上就是原文件，属"事实修正"而非"编辑"，没有可反悔的用户意图。新路径随下次保存落盘；
- **由此产生的不对称是设计内行为**：在对话框里手动定位 A 并级联找回 B、C 后按 Ctrl+Z，只有 A 回退到缺失状态，B、C 保持找回的新路径——撤销的只是"给 A 选了这个文件"的决定，B、C 的找回有独立的 hash 校验背书，不依赖该决定；
- 撤销后资源检查对话框**不会重新弹出**（它只在工程打开时弹一次），回退为缺失的 clip 通过右键菜单「重新定位文件…」修复（§3.4）；
- 「保持缺失」不修改工程；「确认」待确认项（无 sha512 的按文件名命中，§2.3）只转正瞬态状态并补算 sha512，不进 undo 栈。

待讨论：重映射是否需要提供"仅本次会话生效"模式（工程文件不动，适合只是帮别人看工程的场景），见 §6。

### 3.4 音频 clip 右键菜单：重新定位文件

除对话框外，音频 clip 的右键菜单（`TracksGraphicsView` 中已有音频 clip 专属菜单项，如 Extract MIDI Score）新增「**重新定位文件…**」（Relink File…）：

- **常驻显示**，不只在缺失时出现——同一个入口覆盖两种场景：文件缺失时的修复（对应 DiffScope 的 replaceAudioClip），以及文件正常时用户主动换源（如换成新一版的伴奏导出）；
- 选择新文件后：更新路径信息（含重算 sha512，见 §2.1）、重新解码波形，作为**一次可撤销的工程编辑**落盘；
- clip 处于缺失状态时，该项作为菜单**首项**并加粗/带警示图标，就近提供最快修复路径；缺失 clip 较多时，用户更适合走资源检查对话框的级联解析（§3.3），两个入口互补：右键菜单管"这一个"，对话框管"这一批"；
- 换源后 clip 的 `clipStart` / `clipLen` 若超出新文件长度，按现有的越界裁剪逻辑处理并提示。

## 4. 方案分层与落地顺序

1. **方案 1（基础设施）**：`AudioClip` 数据结构扩展 + §2.2 序列化布局（含未知字段 round-trip）+ §2.3 多级解析 + 后台 hash 计算。兼容性：`path` 语义不变，旧工程与其他编辑器完全兼容；无 workspace 附加信息的旧工程退化为现状行为；
2. **方案 3（容错层）**：clip/轨道离线状态 + §3 分级交互 + 资源检查对话框骨架与音频页（歌手页以后追加，见 §3.3）；
   右键菜单「重新定位文件…」（§3.4）属于这一层，可与对话框共用同一个换源实现；
3. **方案 2（资产管理）**：导入音频时提供「复制到工程资产文件夹」选项（可设默认，目录如 `assets/audio/`，复制后自动获得 `relativeDir`）+ 「整理工程 / 收集所有音频」菜单项（对应 Cubase Back up Project、Reaper Save project + move media 工作流）；
4. **方案 4（可选，视需求排期）**：导出为压缩包（zip 打包 .dspx + 资产目录），"一个文件发出去"的分享体验，不需要改动 opendspx 规范。

## 5. 与 DiffScope 的差异清单

| 行为 | DiffScope | ds-editor-lite（本方案） |
| --- | --- | --- |
| FileContentChanged | 警告通知 | 静默接受，重算 hash |
| FileMoved（relativeDir 命中） | 警告通知 | 完全静默 |
| FileMoved（同级兜底命中） | 警告通知 | 汇总一条非阻断通知 |
| workspace 字段 | relativeDir / formatEntryClassName / userData / sha512 | 另写 absoluteDir / fileName（DiffScope 忽略），不写 talcs 两字段但保留 round-trip |
| 缺失处理范围 | 音频 | 音频（v1）+ 歌手/声线包（后续，同一对话框加页） |

## 6. 剩余待讨论的问题

1. **schema 演进沟通**：`workspace["diffscope"]["audio"]` 的字段所有权在 DiffScope 侧，我们额外写入 `absoluteDir` / `fileName` 需与其确认未来不冲突（当前其 reader 会忽略未知字段，无碍）；
2. **歌手重映射的持久化语义**：默认「写回工程 + 可撤销」，是否额外提供「仅本次会话生效」模式？
3. **包管理器集成程度**：`packageId` 是否足以定位到可下载的包源（决定「安装对应包」按钮能否实现）；
4. 资产目录的命名与位置约定：`assets/` vs `<工程名>_assets/` vs 用户可配置（影响 `relativeDir` 稳定性，建议定死默认值）；
5. 导入时「复制到资产文件夹」默认开启还是每次询问；工程未保存时如何处理（先缓存临时区，保存时落位？）；
6. hash 计算开销：全文件 SHA-512 对长录音的导入/校验耗时是否可接受；需后台任务 + 缓存（按 mtime + size 判断是否重算）；
7. 峰值缓存（`AudioInfoModel::peakCache`，现有 TODO）是否顺势移出工程文件，避免膨胀。
