# 工程音频资产跨设备可移植性方案

> 状态：讨论稿（2026-07-17）
> 问题：创作者导入音频后分享工程（附带音频文件），另一台设备上打开时因路径差异无法找到音频。

## 1. 问题根源

当前实现中，音频路径的处理链路如下：

- `AudioClip` 只保存一个路径字符串 `m_path`（`src/app/Model/AppModel/AudioClip.h`），导入时直接写入用户选择的**绝对路径**；
- 保存工程时原样序列化进 .dspx 的 `path` 字段（`DspxProjectConverter.cpp` 序列化处），加载时原样读回；
- 解码失败时没有"缺失媒体"的概念：clip 不会进入可恢复的离线状态，也没有重新定位（relink）的入口。

因此只要换一台机器（或移动音频文件），绝对路径必然失效，且用户没有任何补救手段。

这实际上是三个独立的子问题：

| 子问题 | 说明 |
| --- | --- |
| 存什么路径 | 绝对路径不可移植，需要相对路径或其他定位手段 |
| 资产放在哪 | 音频散落在任意目录（如下载目录），无法随工程一起分发 |
| 丢了怎么办 | 找不到文件时的容错与恢复体验 |

DAW 行业对此有成熟套路（Cubase「Back up Project」、Reaper「Save project + move media」、Premiere/Ableton 的缺失媒体重定位对话框），以下方案参考这些实践，按层次递进。

## 2. 方案分层

### 方案 1：路径解析层——相对路径优先 + 兜底搜索

**目标**：改动最小，为后续方案提供路径解析基础设施。

- **保存时**：若音频文件与工程文件同盘且位于工程目录附近，写入相对于 .dspx 文件的**相对路径**；否则仍写绝对路径。
- **加载时**按以下顺序解析：
  1. 路径本身可用 → 直接使用（相对路径基于工程文件所在目录解析）；
  2. 绝对路径失效 → 提取文件名，在工程文件所在目录及其子目录（如 `assets/`）中按文件名搜索；
  3. 均失败 → 进入缺失状态（见方案 3）。
- 在 clip 的 `workspace`（编辑器私有扩展区，已有机制）中额外保存**文件 hash + 文件大小**，兜底搜索命中时校验，避免同名不同文件误匹配。

**兼容性**：不改变 dspx 规范语义（`path` 字段本来就是字符串），旧工程完全兼容。

### 方案 2：资产管理层——导入时收集到工程资产目录（核心）

**目标**：资产随工程走，分享时拷贝「工程文件 + 资产目录」即可在任何机器上打开。

- 工程已保存的前提下，导入音频时提供选项（可设为默认行为，进入首选项）：「**复制到工程资产文件夹**」，目录结构如 `<工程名>_assets/audio/` 或工程同级 `assets/`；
- 复制后 clip 指向资产目录内的相对路径；
- 对已有工程提供菜单项「**整理工程 / 收集所有音频**」：一键把散落在各处的音频收拢进资产目录并改写路径（对应 Cubase Back up Project、Reaper Save project + move media 的工作流）。

**依赖**：方案 1 的相对路径解析。

### 方案 3：容错层——缺失媒体的用户体验

**目标**：任何情况下工程都能打开，不静默丢数据，用户有明确的恢复手段。

- 音频找不到时 clip 进入**离线状态**：轨道上显示占位（灰色 + 「文件缺失」标记），工程照常打开，clip 的时值/增益等信息完整保留；
- 打开工程时如有缺失，弹出汇总对话框：「N 个音频文件缺失」，支持**手动重新定位**；
- 用户定位到其中一个文件后，自动以该文件所在目录尝试解析其余缺失文件（Premiere/Ableton 的经典做法）；
- 重新定位成功后更新 clip 路径并触发重新解码。

**说明**：无论方案 1/2 是否实施，这一层都值得单独存在。

### 方案 4（可选）：打包分享格式

**目标**：「一个文件发出去」的分享体验。

- 增加**导出为压缩包**功能：zip 容器打包 .dspx + 资产目录；
- 接收方打开时解包到本地目录；
- 属于编辑器侧的导出功能，**不需要改动 opendspx 规范**。

## 3. 推荐落地顺序

**方案 2 + 3 是治本**：资产随工程走是所有 DAW 验证过的模型；缺失兜底保证工程永远打得开。方案 1 是两者的基础设施，应最先实施。

1. 方案 1：路径解析函数（保存转相对 / 加载多级解析）+ workspace 中记录 hash；
2. 方案 3：离线 clip 状态 + 缺失媒体汇总对话框 + relink；
3. 方案 2：导入时复制选项 + 「整理工程」菜单项；
4. 方案 4：视需求排期。

## 4. 待讨论的问题

1. **dspx 规范与生态兼容**：dspx 是 OpenVPI 生态的交换格式，其他编辑器也会读 `path` 字段。**把相对路径写进 `path` 是否符合规范/生态预期？**
   - 若规范只认绝对路径：相对路径与 hash 全部放 `workspace` 扩展区，`path` 保持绝对路径作为兼容值；
   - 若规范允许相对路径（或可以推动约定）：直接写相对路径，可移植性对所有编辑器生效。
2. 资产目录的命名与位置约定：`<工程名>_assets/` vs 工程同级 `assets/` vs 用户可配置；
3. 导入时「复制到资产文件夹」是默认开启还是每次询问；工程未保存时如何处理（先缓存到临时区，保存工程时再落位？）；
4. hash 算法与计算时机（大文件全量 hash 的开销，是否只 hash 头部 + 大小）；
5. 峰值缓存（`AudioInfoModel::peakCache`，现有 TODO：将峰值数据保存到其他地方）是否顺势移入资产目录旁的缓存文件，避免工程文件膨胀。
